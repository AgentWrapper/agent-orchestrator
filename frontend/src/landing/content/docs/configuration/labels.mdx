---
title: Issue labels
description: Use informational status labels and route assigned AO tickets to worker harnesses.
---

Labels never authorize or veto tracker intake. Assignment is the sole admission
signal: assigned issues are eligible, unassigned issues are inert. Park an issue
by unassigning it.

## Status labels

These labels are informational. They do not change daemon admission.

| Label           | Meaning                            |
| --------------- | ---------------------------------- |
| `deferred`      | Deferred for future consideration. |
| `charter`       | Charter-managed work.              |
| `charter-audit` | Charter audit work.                |
| `human-review`  | Human review requested.            |

## Agent routing labels

Routing labels select a worker harness only after assignment admits the ticket.
Every routed worker consumes normal project capacity, and a pinned harness does
not substitute to another harness if unavailable.

| Label              | Harness                                  |
| ------------------ | ---------------------------------------- |
| `agent:codex`      | `codex`                                  |
| `agent:fugu`       | `codex-fugu`                             |
| `agent:codex-fugu` | Accepted optional alias for `codex-fugu` |
| `agent:claude`     | `claude-code`                            |

There is no label-based capacity escape. Legacy `nopool` labels are ignored for
admission and capacity accounting.
