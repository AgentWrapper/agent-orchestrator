Linear issue intake

Phase 1 plan for the first working Linear intake path on top of feat/github-issue-intake-v2.

Summary

Area Decision
Working scope Ship the full Linear intake workflow with minimal UI: a project can select Linear, choose one accessible Linear team, save config, and let the daemon spawn workers from assigned open issues.
Provider rule Each project connects exactly one tracker provider at a time: either GitHub or Linear. The config and UI switch provider modes; they do not merge filters across providers.
Scope rule Linear stores the selected team's teamId, not git-origin repo inference. GitHub keeps repo; Linear ignores it.
Credential Linear intake uses one Linear API key available to the daemon via LINEAR_API_KEY.

Architecture design

Phase 1 is end-to-end but minimal. Electron only exposes the controls needed to enable Linear intake, including a team dropdown populated from Linear's official teams query for the authenticated account. The daemon reads project config, resolves exactly one tracker adapter for the selected provider, asks Linear for assigned spawnable work, and spawns sessions through the existing session service. Label filters, previews, and richer picker flows are follow-up work.

Minimal UI

The phase-1 tracker intake card should expose only the controls required to enable the workflow: enable intake, choose a provider from the standard select, choose a Linear team from the standard select, and save through the existing settings form action. No labels, preview count, team search, or advanced filters.

Work plan

Goal A project can opt into provider=linear, choose an accessible Linear team from the UI, save the config, and have the daemon poll Linear for spawnable issues assigned to the authenticated Linear user. A project uses one provider mode at a time; enabling Linear intake replaces GitHub intake for that project.

What changes

  • Add Linear as the second tracker provider while keeping one active provider per project.
  • Add a minimal settings path: select Linear, choose an accessible team from a dropdown, and save.
  • Use Linear GraphQL for authenticated user, teams, issue fetch, and assigned spawnable issue listing.
  • Reuse the existing intake loop for polling, duplicate suppression, prompts, worker spawning, and backoff.

Why

This keeps the daemon as the tracker boundary, preserves the one-provider-per-project model, and proves Linear intake with the smallest useful end-to-end slice before label filters, preview, and richer provider polish.

Use viewer to identify the authenticated user, teams to populate the team dropdown, an issues query with filter variables for team, assignee, and spawnable workflow state types, orderBy: updatedAt, and Relay cursor pagination for result sets. Store linear:<identifier> when Linear returns an identifier such as ENG-142.

Rate-limit rule: push team, assignee, and spawnable-state filtering into GraphQL. Linear GraphQL does not provide a GitHub-style ETag flow for this use case, so AO uses Linear rate-limit reset headers to extend per-project backoff when Linear returns 429. Keep AO's observer cadence; do not add webhooks yet.

Future scope

  • Label filters and matching preview counts.
  • Webhooks or Linear issue updates once AO needs realtime tracker sync.
