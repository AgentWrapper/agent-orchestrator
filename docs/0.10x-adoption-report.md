# ao 0.10x Adoption Report

Investigation report, 2026-07-05. Scope: (1) learn the upstream 0.10x tree, (2) audit
every deliberate change in our abandoned 0.9x fork against 0.10x, (3) mine the retired
`ao-ops` daemons for workflow intent and map that intent onto *vanilla* ao 0.10x +
polypowers, (4) propose vanilla setup steps for Phase 6 of the consolidation plan.
No code was changed anywhere; ao stays vanilla.

Source trees examined:

- This repo — upstream ao 0.10x (Go daemon + Electron supervisor), at `4ee51fe0`.
- `oldao/ao-src` — our 0.9x fork (HEAD `76ad9579`). Merge-base with upstream `main`
  is `5897b4e8` (#2109); **36 fork-only commits**. Same repo lineage — upstream
  rewrote the backend in Go in-repo, so a real git comparison was possible.
- `oldao/ao-ops` — retired deterministic daemons (intent source only).
- `agent-vault` `origin/main`: `docs/ao-polypowers-briefing.md`,
  `docs/consolidation-plan.md` (Phase 6). Read only; not modified.

## 1. Executive summary

**Verdict: vanilla 0.10x is viable for the polypowers workflow with zero patches.**
Every fork change is either already covered by the Go rewrite, obsoleted by its
architecture, achievable with plain configuration, belongs in the vault/nickify layer,
or is a genuine upstream gap worth *filing as an issue* rather than re-patching.
Nothing *requires* a local patch. One deliberate exception was later decided
(2026-07-05): per-session `--model` gets implemented in our tree as an
upstream-shaped change (§7 filing list, item 1).

The load-bearing integration facts, verified in code:

1. **Worker env is pure configuration.** `ProjectConfig.Env`
   (`backend/internal/domain/projectconfig.go:24-27`) is forwarded verbatim into every
   spawned session (`backend/internal/session_manager/manager.go:1138-1200`). So
   `POLYPOWERS_AUTOMERGE=1` and `POLYPOWERS_REPO=owner/repo` are one
   `ao project set-config <id> --env …` away. Caveat: env is **project-scoped** — no
   per-role or per-session env; workers, orchestrators, and reviewers in a project all
   get the same map. Caveat 2: `set-config` **replaces** the whole config blob, it does
   not merge.
2. **0.10x has no auto-merge — and that is correct for us.** Merge is a manual HTTP
   endpoint (`POST /api/v1/prs/{id}/merge`); lifecycle "reactions" only *nudge* the
   agent (CI failing, changes requested, conflicts) and emit notifications
   (`backend/internal/lifecycle/reactions.go`). The polypowers contract puts the
   gated merge+deploy loop *inside the worker* (`/address-issue-queue --merge` under
   `POLYPOWERS_AUTOMERGE=1`), so ao's lack of merge machinery is a feature: no
   overlap, no fight.
3. **Skills need nothing from ao.** The vault's account-level seed + `sx install
   --hook-mode` SessionStart hook keeps skills fresh in `~/.claude/`; ao workspaces
   never need skill installs. Our fork's "install SDLC skills via sx into worker
   workspaces" commits are fully superseded by the vault mechanism (ao's `postCreate`
   hook exists as a fallback lever but should not be needed).
4. **The old ao-ops daemon fleet is ~80% absorbed** — partly by 0.10x's rewrite
   (always-on daemon, conservative reaper, CDC push), mostly by the polypowers skills
   (gate, cleanup, deploy, second-signer review). What remains are a handful of real
   upstream gaps (§4) and a few workflow decisions (§6).

Four candidate upstream issues (details in §3.2): no per-session `--model` on spawn,
no orphaned-worktree self-heal on create, tmux child-process leak on kill, and no
doctor probe for the reactions pipeline. A fifth (ambient MCP inheritance by
workers/reviewer) was evaluated and **dropped by decision D-f**: in our
single-account design, workers inheriting the account's full environment is the
point, not a bug. Two behaviors to verify empirically before filing anything:
multiline `ao send` reliability, and respawn-onto-stale-branch semantics.

## 2. The 0.10x codebase in brief

One always-on **Go daemon** (loopback-only HTTP on `127.0.0.1:${AO_PORT:-3001}`:
REST + SSE + terminal WebSocket `/mux`), a **thin Cobra CLI** that is purely an HTTP
client, and an **Electron supervisor** UI. State lives in SQLite under
`~/.ao/data` (`AO_DATA_DIR`), with `~/.ao/running.json` (`AO_RUN_FILE`) as the
PID/port handshake. DB triggers feed a change log fanned out over SSE.

- **Spawn pipeline** (`backend/internal/session_manager/manager.go`): session row →
  git worktree (`adapters/workspace/gitworktree`) → provision (project `symlinks` +
  `postCreate`) → tmux runtime → agent argv from the adapter → hooks installed into
  the agent's workspace config. 23 agent adapters (claude-code, codex, opencode, …)
  with an agent catalog + auth preflight on spawn (PR #2411).
- **Tracking**: durable facts only (activity state from agent hooks via
  `ao hooks <agent> <event>`; PR facts from a 30s GitHub SCM observer); display
  status derived at read time. The reaper probes runtimes every 5s and is
  deliberately conservative — a failed probe is never treated as death, and
  termination requires dead runtime + dead process + no recent activity.
- **Lifecycle reactions** (`backend/internal/lifecycle/reactions.go`): nudge the
  agent on CI failure / changes-requested / merge conflict (max 3, deduped,
  restart-safe), terminate the session once its PR merges, emit `ready_to_merge` /
  `pr_merged` / `needs_input` notifications. Notification ≠ action: **ao never
  merges.**
- **Configuration**: the daemon takes *no config file* — global settings are env vars
  only (`AO_PORT`, `AO_DATA_DIR`, `AO_AGENT`, `AO_ALLOWED_ORIGINS`, telemetry;
  deliberately no `AO_HOST` — the legacy `agent-orchestrator.yaml` is gone). Per-project
  config is a typed JSON blob in SQLite, set via `ao project add` / `ao project
  set-config` (`PUT /api/v1/projects/{id}/config`): `defaultBranch`, `sessionPrefix`,
  `env{}`, `symlinks[]`, `postCreate[]`, `agentConfig{model, permissions}`,
  `worker`/`orchestrator` role overrides (harness + agentConfig only, **not** env),
  `reviewers[]`, `trackerIntake{}`.
- **Dispatch**: `ao spawn --project P --prompt "…" [--agent H] [--branch B]
  [--model M] [--issue N] [--claim-pr]` and `ao send --session S --message "…"`.
  Model precedence is per-spawn `--model` over role `agentConfig.model` over
  project `agentConfig.model` over the adapter default. Orchestrator spawn is
  HTTP-only for now (`POST /api/v1/orchestrators`). External drivers watch
  `ao session ls --json` or SSE `GET /api/v1/events`.
- **Still in flight** (`docs/STATUS.md`): `trackerIntake` is configured but **not
  driven by any runtime loop yet** (label-driven intake is a no-op today, upstream
  issue #112); merge/review actions are HTTP-only with no CLI parity; `ao events`
  CLI not built; GUI-launch login-shell env resolution still "proposed".

### 3.2 Candidate upstream issues (the only surviving fork value)

1. **Ambient MCP inheritance — dropped by decision D-f (2026-07-05).** No MCP
   scoping exists in the Go backend; workers and the nominally read-only reviewer
   inherit the account's full global MCP server set (fork `f39e0a7d`; upstream-TS
   #2232 intent unrealized in the rewrite). For a multi-tenant deployment that
   would be a security issue, but our design is the opposite: one account, workers
   deliberately inherit everything (skills, hooks, auth, MCP). Not filing; noted
   here so nobody re-flags it.
2. **No per-session `--model` on spawn** (fork `926297c9`; upstream #2117). Role and
   project-level model config is only a partial substitute — a mixed-cost pool
   (e.g. haiku deploy workers) can't be spawned per-session.
3. **No orphaned-worktree self-heal** (fork `c1cc25c1`; upstream #1651 regressed in
   the rewrite): no `git worktree prune` on create, stale registrations are handed
   back as live workspaces, checked-out-elsewhere is a hard 409. This caused the
   worst ao-ops incident (280-cycle silent respawn thrash), so it matters at fleet
   scale.
4. **tmux child-process leak on kill** (fork `2f4ec8c0`; upstream #1369 same class):
   `Destroy` never reaps processes that detached from the pane's process group.
5. **`ao doctor` can't assert the reactions pipeline** (fork `9a0955ba`): doctor
   validates the GitHub token and hooks log but nothing exercises
   SCM-observer → nudge. Low severity.

Two VERIFY items to test during standup, filing issues only if they reproduce:
multiline `ao send` submission under a busy agent TUI (`3db1d946`), and
respawn-onto-existing-stale-branch reuse semantics (`86a9de28`).

## 4. ao-ops: intent, and how vanilla ao + polypowers express it

ao-ops was six systemd poll daemons + host patches wrapping a pinned ao 0.9.5 to run
an autonomous label-driven SDLC over four repos. It reinvented ao and predated the
generic polypowers bundle (it never set any `POLYPOWERS_*` var — only `AO_*`). Its
value is the requirements it encodes. Each requirement below maps to how the vanilla
stack expresses it — **skills-layer** (polypowers), **ao-config**, **ao-builtin**,
**gap**, or **retired**.

| # | Requirement (from ao-ops) | Vanilla expression |
|---|---|---|
| R1 | Unattended intake: a GitHub label (`agent:backlog`) is the "go" signal; claim-before-spawn, one worker per issue cluster, concurrency caps, natural-language spawn directive | **Skills-layer + gap.** The polypowers answer is `/address-issue-queue` — dispatch a *batch* to one worker via `ao spawn --prompt "/address-issue-queue <ids> --merge"`. Continuous label-driven intake maps to ao's `trackerIntake`, which is configured but **not runtime-live yet** (upstream #112) — until it lands, intake is manual dispatch or a trivial external cron that shells `ao spawn` (allowed: cron drives ao, it doesn't modify it) |
| R2 | Deterministic agent/model mix per issue (60/20/20 codex/claude/opencode) for cost control; `model:<id>` label pins | **ao-config (partial).** `--agent` per spawn covers the harness mix; per-session model is gap §3.2-2. Role-level model config covers "deploy workers are cheap" only if deploys run as a distinct project/role |
| R3 | Resume-don't-respawn; anti-thrash (failure counter → park); preserve unpushed commits from orphaned worktrees | **ao-builtin (partial).** `ao session restore` + conservative reaper cover resume; orphan-worktree self-heal is gap §3.2-3. The park-with-reason behavior now lives in the skills (`/address-issue-queue` parks with a written reason) |
| R4 | Deterministic merge gate: CI green + review attestation + no sensitive path + known work type; else park; squash only; TOCTOU-pinned | **Skills-layer.** The polypowers gate (final-review clean + CI green + threads resolved) enforced *inside* the worker under `POLYPOWERS_AUTOMERGE=1`. Sensitive-path rules move to repo fragments (repo-specific behavior lives in the repo, never in ao). ao contributes the `ready_to_merge` notification only |
| R5 | Independent second-signer, cross-family review, fail-closed | **Skills-layer.** `/final-review` cross-family review is the successor. ao's own `reviewers[]` config (read-only reviewer) is available as an *additional* layer if wanted |
| R6 | Auto-sail grace window: announce, wait one cycle, `hold` label = operator veto | **Retired by decision D-c.** Default is merge on gate-pass; no veto window. Sensitive changes are marked in repo fragments and park instead (path-level marking is a vault-side spec item, see D-c) |
| R7 | Post-merge housekeeping regardless of who merged (branches, worktrees, beads, zombie sessions) | **Skills-layer + ao-builtin.** `/cleanup-merge` does the repo/bead side; ao terminates the session when its PR merges and `ao session cleanup` exists for stragglers |
| R8 | Recover dead workers stranding green-but-unsigned PRs (nudge, or respawn `--claim-pr`) | **ao-builtin (partial).** Reactions nudge *live* sessions; `ao spawn --claim-pr` exists for manual adoption of a stranded PR. Nothing automatically respawns a dead worker — acceptable at current scale; revisit after standup |
| R9 | Serialized, coalesced, cheap-model deploy pool driven by `agent:deploy` label | **Skills-layer (changed shape).** In autonomous mode every queue item ends deploy-verified (`/deploy-verify` runs in the same worker), so a separate deploy pool is obsolete as a *mechanism*; per decision D-d the cheap-model (haiku) deploy-worker optimization survives as the use case for the per-session `--model` upstream issue (§3.2-2) |
| R10 | Durable fleet supervision surviving restart (`ao start` forgot its supervised set) | **ao-builtin.** Root cause rearchitected away: one always-on daemon supervises everything. Boot persistence = start the daemon at login (systemd user unit *invoking* vanilla `ao`, which is fine) |
| R11 | Worker↔dashboard identity (title conversations `cc-N · #issue slug`) | **Retired mostly.** ao names sessions (`--name`, `sessionPrefix`, `ao session rename`); the old implementation poked agents' private state files and should not return. If cross-surface naming still matters, put self-titling in the spawn prompt |
| R12 | One-command idempotent project onboarding (config + labels) | **ao-config.** `ao project add` + `ao project set-config` (two commands); label taxonomy — if still used — belongs to nickify repo scope |
| R13 | Two-source registry, inert-by-default, control-plane paths protected everywhere | **Retired.** Existed to coordinate six external daemons. With no external daemons there is nothing to gate; sensitive paths live in repo fragments |
| R14 | Operator outcome events + "while you were away" digest (`ao whatsnew`) | **Partial gap.** ao has durable notifications (`ready_to_merge`, `pr_merged`, `needs_input`) and an SSE event stream, but no digest CLI (`ao events` not built yet). Watch upstream; don't rebuild |
| R15 | Notification fan-out (Slack, desktop, urgency routes) | **Gap at the ao layer.** 0.10x notifications render in the Electron app; there is no notifier/Slack config (the fork-era `notifiers.slack` yaml is gone). Slack is required (decision D-b): a small external, read-only SSE→Slack notifier is the vanilla-compatible shape — build at standup, keep it decision-free |
| R16 | Host patches (`--model` spawn flag; loopback WS bind) | **Retired.** Loopback is now the hardcoded default; `--model` is upstream issue §3.2-2 |
| R17 | Remote dashboard via Tailscale Serve | **Works vanilla (verified in practice).** The renderer runs in a plain browser: `frontend` `npm run dev:web` (`VITE_NO_ELECTRON=1 vite`), whose dev server proxies `/api` and `/mux` to the daemon (`frontend/vite.renderer.config.ts:59-70`, target overridable via `AO_DEV_API_TARGET`; direct cross-origin access is also possible via `AO_ALLOWED_ORIGINS` + `VITE_AO_API_BASE_URL`). Tailscale Serve fronting the vite port is exactly the 0.9x setup minus the loopback patch, which is now the default. Caveats: it is a dev-server arrangement, not a packaged web build; there is still no auth (the tailnet is the security boundary; `/mux` skips origin checks); Electron-only surfaces are stubbed in browser mode (daemon start/stop supervision, native notifications, the inspector Browser tab, updates — `frontend/src/renderer/lib/bridge.ts`); and `VITE_NO_ELECTRON=1` flips three hooks to preview mode — the workspaces query returns **mock data** and per-session PR summaries are **disabled** (`useWorkspaceQuery.ts:28-33`, `useSessionScmSummary.ts:10,24`) — so the web view is slightly degraded vs Electron |

## 5. The workflow contract and proposed vanilla setup

From `agent-vault/docs/ao-polypowers-briefing.md` (contract stable through the flip)
and consolidation-plan Phase 6. Guiding principle restated: **if ao seems to need a
change, that is a smell — the fix belongs on the nickify side.** Nothing found in
this investigation violates that principle.

What ao must guarantee per worker session, and how vanilla 0.10x satisfies it:

| Contract item | Vanilla 0.10x mechanism |
|---|---|
| Working dir = git worktree of the target project | Default spawn behavior (worktree per session) |
| `POLYPOWERS_AUTOMERGE=1` grants the merge+deploy loop | `ao project set-config <id> --env POLYPOWERS_AUTOMERGE=1` |
| `POLYPOWERS_REPO=owner/repo` optionally pins the repo | Same `--env` mechanism; usually unnecessary (worktree remote → `gh repo view` works) |
| `gh` auth + vault SSH per account | Account provisioning (seed + `/nickify` user scope) — outside ao |
| Skills present and fresh | Vault SessionStart hook (`sx install --hook-mode`) — ao uninvolved |
| Beads attachment | Repo config (nickify repo scope) — ao uninvolved |
| Repo-specific rules (rosters, sensitive paths, deploy targets) | Repo fragments via polyscribe — ao uninvolved |
| Unattended operation without permission prompts | Project `agentConfig.permissions` = `bypass-permissions` for autonomous projects (decision D-e) |

Proposed setup sequence (post-flip; **do not execute before Phase 3 lands** — before
the flip ao would drive the coachclaw-coupled skills):

1. Build/install ao from this tree per its own docs (`backend: go build ./...`;
   `go run ./cmd/ao start` or the packaged binaries under `packages/`). Record the
   actual steps in `docs/ao-standup.md` per the plan.
2. Provision the single daemon account once (decision D-a: one daemon, one OS
   account, ao's native model): `agent-vault/install.sh`, then `/nickify` (user
   scope). Authenticate both `claude` and `codex` CLIs in that account. Record in
   `docs/fleet-accounts.md`.
3. Register coachclaw:
   `ao project add --path <coachclaw checkout> --worker-agent claude-code`
4. Configure it in one shot (remember: `set-config` replaces, so pass everything):
   `ao project set-config coachclaw --default-branch main --worker-agent claude-code
   --permission bypass-permissions --env POLYPOWERS_AUTOMERGE=1
   --env POLYPOWERS_REPO=polymath-ventures/coachclaw`
5. Verification run (plan Phase 6.5): dispatch a trivial item —
   `ao spawn --project coachclaw --prompt "/address-issue-queue <id> --merge"` —
   confirm issue → PR → gate → merge → cleanup → deploy-verify with the bead visible
   from Nick's account. Then the negative test T11: a failing gate must PARK, not
   merge.
6. During that run, exercise the two VERIFY items from §3.2 (multiline `ao send`,
   respawn-onto-existing-branch).

## 6. Decisions (recorded 2026-07-05, from Nick)

Originally open questions; answered the same day the report was written.

- **D-a (topology): follow ao's native model — one daemon, one OS account.** ao's
  default is a single always-on daemon spawning all sessions as its own user; the
  multi-account fleet (nhod-claude, nhod-codex, …) was the *old* setup's shape, not
  ao's. Vanilla wins unless a damned good reason appears. Consequence: only one
  account needs seed + `/nickify` (user scope); both `claude` and `codex` CLIs
  authenticate side by side in that one account.
- **D-b (alerting): Slack is required, and the primary UI is the web interface,
  not Electron** (Tailscale Serve → `dev:web`, per §4 R17). Since the Electron
  notification surface is stubbed in web mode, a small **external SSE→Slack
  notifier** is needed: a dumb read-only consumer of `GET /api/v1/events` that posts
  `needs_input` / `ready_to_merge` / `pr_merged` (and park events surfaced by
  workers) to Slack. It reads the API and changes nothing in ao — vanilla-compatible
  glue, to be built at standup time, kept decision-free (no workflow logic in it).
- **D-c (merge policy): default is merge.** No operator grace/veto window. The
  nickified SDLC is trusted to produce mergeable PRs; when the polypowers gate
  passes (final-review clean + CI green + threads resolved), the worker merges.
  Sensitive changes are the exception: they must be *marked* sensitive and those
  park instead of auto-merging. Mechanism: sensitivity rules live in the repo's
  fragments (the briefing already allows fragments to forbid autonomous merge
  repo-wide). **Finding for the vault side:** ao-ops had *path-level* sensitivity
  (`sensitivePrefixes`/regexes per project); the generic polypowers contract only
  documents repo-level forbid. Path-level "sensitive → park" should be specified in
  the fragments/skills (e.g. carry coachclaw's old sensitive list into its
  fragments). Not an ao concern.
- **D-d (deploy): generalized, any repo — when something deployable is ready, it
  deploys.** In autonomous mode this is inherent: every queue item ends
  deploy-verified. Desired optimization: deploy work should run on **low-cost
  (haiku-class) workers across all platforms.** Vanilla ao today can't set a model
  per session (§3.2-2) — the cheap-deploy pool is the concrete use case that makes
  the per-session `--model` upstream issue worth filing. Interim vanilla options if
  needed: a parallel project registration with `agentConfig.model` set to a cheap
  model, used only for deploy dispatches; or accept inline deploy-verify in the
  main worker until upstream lands `--model`.
- **D-e (permissions): `bypass-permissions`** for autonomous projects. Unattended
  workers must never stall on prompts.
- **D-f (upstream filings): the MCP ambient-inheritance issue is dropped** —
  "workers inherit everything; that is the whole point." Single-account by design;
  inheritance is the delivery mechanism for skills, auth, and tools, not a leak.
  The remaining four issues (§3.2: per-session `--model`, worktree self-heal, tmux
  straggler reaping, doctor reactions probe) get filed after the Phase 6
  verification run, which also settles the two VERIFY items. Two *feature asks*
  join them: a runtime loop for `trackerIntake` (upstream #112 — GH-issue pickup is
  the one piece of Nick's target loop ao doesn't run yet) and first-class real-data
  browser mode (see §7).

## 7. Target operating model (Nick's desiderata, 2026-07-05; revised same day)

ao-ops is disowned as a reference ("nothing ever worked there"). The operating model
is ao's own, and it is simpler than anything ao-ops built: **one orchestrator
session per project is the coordinator.** This is not an interpretation — ao bakes
it into every orchestrator's standing prompt: "You are the human-facing coordinator
for project <id>. … Spawn worker sessions for implementation with `ao spawn
--project <id> --name … --prompt …` … Message workers with `ao send` … track their
progress, synthesize their results" (`backend/internal/session_manager/
manager.go:1097-1113`). Workers are likewise told how to reach their orchestrator
for blockers. There is no cron, no external daemon, no "dispatcher" layer: every
policy below is a standing instruction to the orchestrator (its spawn prompt plus
the repo's fragments).

How the orchestrator comes to exist (verified in code, 2026-07-05): **adding a
project in the UI spawns its orchestrator immediately and drops you into it**
(`frontend/src/renderer/routes/_shell.tsx:115`); thereafter the daemon treats it
as ensure-semantics — ask for the project's orchestrator and it returns the live
one or spawns a fresh one (`backend/internal/service/session/service.go:299`,
"ensure-on-load"), with explicit restart/replacement machinery (retire notice to
the old one, canonical-branch respawn). The daemon supervises it like any session:
same reaper, same activity tracking; and because sessions live in tmux, they
survive a daemon restart — boot-time Reconcile re-adopts them. So operationally:
add project → orchestrator exists; it dies → the UI ensures a new one. No CLI
step exists or is needed.

| Want | How the vanilla stack expresses it |
|---|---|

This is exactly what ao is meant to do; the *one* missing piece of hands-off-ness is upstream #112 (`trackerIntake` has config but no runtime loop), which will eventually let ao itself notice new issues. Until then the orchestrator polls as part of its work loop — the residual gap is only that an idle/dead orchestrator has nothing to wake it, so it may occasionally need a "continue" or a respawn |
| Specify models for workers, and per-ticket model overrides | **ao + upstream follow-up.** Per-project/per-role model config exists, and PR #2 adds per-ticket `ao spawn --model` with precedence over role/project defaults. Upstream filing remains required so the delta can be proposed outside our tree. |

Additional requirements (from the same discussion):

1. **Fleet caps — must-have.** Nothing in ao or the skills limits concurrent
   workers. Two layers: a hard cap in the orchestrator's standing instructions
   (never more than N active workers; check `ao session ls` before spawning), and a
   daemon-enforced max-concurrent-sessions-per-project setting as an upstream
   feature ask — an instruction-level cap alone is soft.
2. **Boot persistence.** systemd user unit running vanilla `ao start` at login.
3. **Queue ordering.** Work from `bd ready` or GH — both acceptable; beads adds
   dependency-aware ordering when present.
4. **The kill switch.** Documented stop-everything move: `ao stop`; revoking merge
   authority = `set-config` without the env var (it replaces the whole blob).
5. **Proactive intake.** Scheduled `/bug-hunt` sweeps feed the queue, dispatched by
   the orchestrator like any other work.
6. **Post-flip sanity check (plain terms).** "Gate" in this report is shorthand for
   the merge preconditions the skills themselves enforce — final-review verdict
   clean, CI green, review threads resolved — nothing beyond what /address-issue
   already does. The consolidation plan's own T11 test (a PR with failing checks
   must park, not merge) is worth re-running after any skill version bump, since a
   regression there is the one failure mode that auto-merges a bad change.
7. **Cost visibility.** Session counts per harness/model in the digest, so the
   percentage mix's spend is observable.
