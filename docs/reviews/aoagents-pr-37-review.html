01 Executive summary

PR #37 replaces the rewrite's in-memory fakeStore with two real outbound adapters: a SQLite internal/storage/sqlite persistence layer satisfying ports.LifecycleStore, and a internal/cdc transactional-outbox → JSONL → durable-consumer → broadcaster pipeline. It is genuinely strong work: the canonical write is a single all-or-nothing transaction (session row + change_log + outbox), the revision column is a real optimistic-concurrency CAS rather than a debug counter, escalation budgets are now durable across restart, and the design is honest about what it stubs (noopNotifier, noopMessenger, empty reaper registry) versus what it refuses to fake (Session Manager, HTTP routes).

The transactional-outbox pattern is the correct backbone and the happy-path is well-tested. My concerns are concentrated at the edges of failure: the in-process Broadcaster calls subscriber callbacks synchronously with no panic recovery, so one bad subscriber can silently kill the entire delivery goroutine; a cross-store durability mismatch exists because the JSONL log is fsync'd per write while SQLite runs synchronous=NORMAL, so a power loss can leave the CDC log ahead of the recovered database; and the janitor's deletion watermark is unsafe for the multi-consumer future the schema already advertises. None of these block the merge of a foundational PR, but they should be tracked before live transports subscribe to the broadcaster.

Versus the existing AO, this is a deliberate architectural upgrade — ACID transactions and a durable change stream replace per-session JSON files and an in-memory reaction map — but the rewrite is still infrastructure without a product: no notifier, no SCM poller, no agent process management, no dashboard, no cross-platform process teardown. The storage substrate is arguably ahead of the mature AO; the surface area around it is far behind.

02 What this PR does

The write path is wired live end-to-end in backend/main.go alongside the Lifecycle Manager and reaper:

LCM.Upsert ──▶ sqlite store (1 tx: sessions + change_log + outbox) ──▶ cdc publisher ─50ms─▶ session-events.jsonl (fsync per line) ──▶ cdc consumer ─100ms─▶ byte cursor, seq-idempotent ──▶ Broadcaster ──▶ [ FE transport — deferred ] janitor ─60s─▶ reclaims acked outbox rows below MIN(consumer offset)

Storage — internal/storage/sqlite

  • Driver & migrations. modernc.org/sqlite (pure-Go, no CGO) for clean 3-OS cross-compile; goose embedded migrations in migrations/0001_init.sql; sqlc-generated typed queries checked in under gen/.
  • Atomic Upsert. upsert.go:26 opens one *sql.Tx, runs the revision-CAS persist, then appends the change_log + outbox rows, and commits — all-or-nothing (upsert.go:43).
  • Revision CAS. upsert.go:48 — insert requires incoming revision 0 and persists 1; update requires the loaded revision to match and bumps +1; zero rows affected ⇒ revision-mismatch error (upsert.go:60,77).
  • Connection policy. WAL + busy_timeout(5000) + foreign_keys(ON) + synchronous(NORMAL) (db.go:24), with SetMaxOpenConns(1) (db.go:44) enforcing a single writer so WAL's single-writer rule cannot be violated by the pool.
  • Metadata side-channel. branch, workspacePath, etc. live in session_metadata written by PatchMetadata (store.go:98) — deliberately off the CDC path: no revision bump, no event.
  • Durable reaction trackers. escalation budgets persist in reaction_trackers (0001_init.sql:89), fixing the in-memory-only behaviour that re-fired human pages after a daemon bounce.

CDC — internal/cdc

  • Publisher (publisher.go:97) drains the outbox to JSONL on a 50 ms ticker; an append failure stops the pass (row marked failed, retried next tick) so ordering is never violated by skipping ahead.
  • JSONL log (jsonl.go:52) appends + Sync()s each line; at 1 MiB it rotates by renaming aside (jsonl.go:86) and writing a reset marker as the new first line.
  • Consumer (consumer.go:114) tails via a byte cursor, detects rotation with os.SameFile / shrink (consumer.go:130), resyncs from a full-state DB snapshot on a gap, and persists its seq in consumer_offsets (consumer.go:157).
  • Janitor (janitor.go:75) reclaims acked outbox rows below the slowest consumer offset; change_log is never touched.
  • Broadcaster (broadcast.go:38) is the in-process fan-out the FE transport will subscribe to; WS/SSE wiring is intentionally deferred.

Composition root

main.go, cdc_wiring.go, and lifecycle_wiring.go stand up the publisher/consumer/janitor/broadcaster and construct the LCM with durable escalation (WithReactionStore) and a session lister (WithSessionLister(store.ListAll)). Three collaborators are honestly stubbed with TODO(daemon-lane) markers: noopNotifier, noopMessenger, and an empty reaper.MapRegistry{} — each safe only because a no-op there drops an outbound side-effect rather than faking success.

03 Motivation

The PR body states the goal plainly:

"Replaces the in-memory fakeStore with the two real outbound adapters the Go backend needs to persist and deliver session state."

The deeper motive is inferable from the rewrite's shape. The existing AO keeps session state as one JSON file per session and holds reaction/escalation budgets in an in-memory map that resets on restart (lifecycle-manager.ts:498). The rewrite is betting that a single embedded database with a real change stream is a better substrate for the same supervisory loop. Two pain points are called out as fixed:

  • Durable escalation. The PR explicitly fixes "the in-memory-only behaviour that re-fired human pages after a daemon bounce" — escalation budgets now survive a restart (reaction_trackers).
  • A live delivery seam. The CDC stream + Broadcaster is positioned as the single seam between persistence and live transports (WS/SSE), so the frontend can be built and swapped without touching the pipeline — a cleaner contract than the existing dashboard's 5 s SSE poll of /api/sessions.

The author is also explicit about engineering discipline: "stub when the no-op is semantically honest; do not stub when the no-op would lie about success." That distinction (notifier vs. Session Manager) is the right one and worth crediting.

04 Correctness assessment

Findings are ranked by severity. Line anchors were verified against the PR head. None are merge-blockers for a foundational, transport-deferred PR — but H1–H3 should be resolved before any live subscriber attaches to the broadcaster.

HighH1Broadcaster has no panic recovery; one bad subscriber silently kills all delivery

Publish (broadcast.go:38) holds the read lock and calls each subscriber callback synchronously in a bare loop. A panic in any fn(e) propagates up through Consumer.Poll into the consumer goroutine and terminates it — after which no further events are delivered to anyone, with no supervisor to restart the loop. The doc comment asks subscribers to "not block," but nothing enforces it: a slow callback (a stalled WS write) stalls the poll loop, delays rotation detection, and backs up the whole pipeline. Wrap each callback in recover(), and give subscribers a bounded buffer with a drop-or-disconnect policy.

HighH2Cross-store durability mismatch: fsync'd JSONL vs. synchronous=NORMAL SQLite

The JSONL log Sync()s every line (jsonl.go:75), but the database runs synchronous(NORMAL) in WAL mode (db.go:27), under which a committed transaction can be rolled back on power loss / OS crash (the DB stays consistent, but the last few commits may vanish). Because the publisher fsyncs the event to JSONL and the consumer persists its offset before that commit is guaranteed durable, a hard crash can leave the CDC log and consumer offset ahead of the recovered change_log. On restart the seq the consumer believes it delivered no longer exists in the DB. It is self-healing only via a rotation-triggered snapshot resync, which may not happen for a long time. Decide this explicitly: either synchronous(FULL) for canonical writes, or treat all canonical state as fully re-derivable and document that the JSONL log is advisory.

HighH3Janitor watermark is unsafe for the multi-consumer model the schema advertises

MinConsumerOffset (cdc_store.go:92) computes MIN(last_seq) only over rows that exist in consumer_offsets. A consumer that has never acked has no row (GetOffset returns 0 without inserting, cdc_store.go:60), so it is invisible to the MIN. The janitor (janitor.go:75) then deletes outbox rows below the other consumers' higher watermark — rows the new/restarted consumer never read. Since outbox rows that predate the current JSONL segment are gone from the log too, that consumer can permanently miss events (it only resyncs on rotation). Only one consumer is wired today, so this is latent — but consumer_offsets, the per-consumer cursor, and the comment all promise multi-consumer support. Register each consumer's offset row at startup (seed at its snapshot seq) before it counts toward the watermark.

MediumM1A corrupt JSONL line is permanently dropped, leaving the FE stale until next rotation

processLines (consumer.go:181) logs and continues on an unmarshalable line, advancing past it forever. A torn or truncated write (or a future schema mismatch) silently drops that session's update; downstream state stays stale until the next event for that session or a rotation-triggered snapshot. SQLite remains the source of truth so it is eventually consistent, but there is no targeted re-fetch. Consider: on parse failure, fall back to a snapshot resync rather than skip.

MediumM2Metadata writes are invisible to the CDC stream

PatchMetadata (store.go:98) is intentionally off the canonical path — no revision bump, no change_log/outbox row. That is a defensible choice, but it means a frontend built purely on the CDC stream will never observe branch, workspacePath, or runtimeHandleId changes; those require a separate read API. The snapshot resync also excludes metadata (payload.Metadata = nil, upsert.go:92). Make sure the FE contract knows metadata is pull-only, or fold a coarse "metadata changed" event onto the stream.

MediumM3Latent referential-integrity gaps (no delete path today, but the schema invites one)

change_log.session_id has no foreign key to sessions (0001_init.sql:56), and outbox.change_log_seq references change_log(seq) with no ON DELETE action (0001_init.sql:67). No code deletes sessions or change-log rows in this PR, so nothing is broken now — but a future retention/cleanup job (which this storage layer will eventually need: change_log grows unbounded) would orphan audit rows or hit an FK RESTRICT. Decide the retention story and encode it in constraints before writing the cleanup job.

LowL1Publisher crash between append and MarkSent re-appends a duplicate line (benign)

If the process dies between log.Append (publisher.go:103) and MarkSent (publisher.go:110), the row stays unsent and is re-appended on restart — producing two JSONL lines with the same seq. The consumer's e.Seq <= c.lastSeq idempotency check (consumer.go:185) drops the duplicate, so this is harmless: correct at-least-once-with-idempotency, exactly as intended. Noting it only to pre-empt a "leak" misreading — the row is not lost, just re-emitted.

LowL2Revision-CAS race is closed only by SetMaxOpenConns(1)

The check-then-write between GetSessionRevision and InsertSession/UpdateSessionCAS (upsert.go:48) is a textbook TOCTOU, defused entirely by the single-connection pool (db.go:44) that serializes all access. The code comments acknowledge this. It is correct today, but the safety lives in one easily-forgotten SetMaxOpenConns(1) line far from the CAS logic; a future pool-size bump for read concurrency would silently reintroduce the race. A conditional UPDATE ... WHERE revision = ? / INSERT ... ON CONFLICT DO NOTHING already makes the write itself atomic — lean on the row-count, not the prior read, as the sole arbiter.

LowL3Byte cursor is in-memory; each restart rescans the whole current segment

Only lastSeq is persisted; c.cursor resets to 0 on restart (consumer.go:45), so the consumer re-reads the entire current segment from byte 0 and dedupes by seq. Correct, and bounded by the 1 MiB rotation cap, so the cost is small — but worth a comment so a future reader doesn't mistake it for a bug.

SolidWhat's done right

The Upsert transaction is genuinely atomic and the defer tx.Rollback()-then-return-Commit() shape is idiomatic Go, not a bug. The revision CAS exactly mirrors the prior in-memory contract and is covered by three focused tests. Ordering is preserved by stopping the drain on first append failure rather than skipping ahead. Rotation-by-rename (not truncate-in-place) is the right call for reliable os.SameFile detection. The stub/no-stub discipline is exemplary.

05 Data-integrity specifics — SQLite + CDC

Schema & migrations

One goose migration (0001_init.sql) with six tables: sessions (typed canonical columns; status never stored), session_metadata (KV, cascade-on-delete), change_log (autoincrement seq — the CDC ordering/idempotency key), outbox (one unsent row per write, partial index WHERE sent = 0), consumer_offsets, and reaction_trackers. Migrations run embedded via goose.Up at Open (db.go:59). There is a clean +goose Down. Gap: only one migration exists, so the forward-compatibility story (additive columns, never-rewrite-history) is unproven — the first schema change will exercise it.

Write ordering & transaction boundaries

The canonical write is correctly a single transaction spanning session + change_log + outbox (upsert.go:26–43), so a reader can never see a session row without its outbox row, and the publisher can never see a half-written event. The outbox sequence is the change_log.seq autoincrement, giving a total order. The publisher drains in seq order and refuses to advance past a failed append, preserving that order downstream.

Durability & WAL

WAL + synchronous=NORMAL is the standard high-throughput SQLite posture and is fine for re-derivable state — but see H2: it is in tension with the per-line-fsync'd JSONL log. There is no explicit checkpoint policy (relying on SQLite's automatic 1000-page WAL checkpoint); under a write-heavy supervisory loop the WAL can grow between checkpoints, and a crash mid-checkpoint is safe but the lost-commit window is real. busy_timeout(5000) + SetMaxOpenConns(1) means writers serialize cleanly and SQLITE_BUSY is effectively impossible for the single writer; readers share the WAL.

Idempotency

Strong on the consumer side: every event carries its seq and the consumer drops anything <= lastSeq (consumer.go:185), so duplicate JSONL lines (L1) and full-segment rescans (L3) are absorbed. The transactional outbox guarantees the DB write and the intent-to-publish commit together, so an event is never published without being persisted. The one asterisk is H2's cross-store ordering under power loss.

Replay / recovery

Recovery is snapshot-based: on a detected rotation gap the consumer rebuilds full current state from store.ListAll (consumer.go:195) and resumes from MAX(change_log.seq). This is a clean, bounded recovery — but it is the only recovery, and it fires only on rotation. There is no mechanism to replay a specific missed seq range, so the corrupt-line case (M1) and the missed-consumer case (H3) wait for the next rotation to self-heal.

Backpressure

This is the weakest area. The Broadcaster fan-out is synchronous and unbounded (broadcast.go:38): there is no per-subscriber queue, no buffer, no drop policy, and no timeout. A slow subscriber applies backpressure all the way to the consumer poll loop, and a panicking one stops it (H1). The publisher's 256-row batch (publisher.go:13) bounds outbox drain per tick, and the janitor bounds outbox growth — but only once a consumer is acking. Until a real transport exists with its own buffering, the backpressure contract is "subscribers must be instant and infallible," which is not a contract a network transport can keep.

08 Major design decisions + alternatives

D1 · Transactional outbox + JSONL CDC, with the DB (not the log) as source of truth

The bet: persist intent-to-publish atomically with the state write, then drain asynchronously; treat the log as a disposable transport and rebuild from a DB snapshot on any gap.

Alternatives: (a) Poll the DB directly from each transport — simpler, no log/janitor/rotation machinery, but loses the clean push seam and re-introduces polling latency. (b) SQLite update_hook / session extension — native CDC, but CGO-only (kills the pure-Go cross-compile) and process-local. (c) Tail the WAL — zero extra writes, but couples you to SQLite internals. Trade-off: the chosen design buys a transport-agnostic seam and durable ordering at the cost of three moving parts (publisher/consumer/janitor) and the cross-store durability question (H2). Reasonable for a product that wants pluggable live transports; heavier than a single-consumer app needs.

D2 · modernc.org/sqlite (pure Go) over mattn/go-sqlite3 (CGO)

The bet: a 3-OS binary that cross-compiles with no C toolchain is worth a performance and maturity trade.

Alternatives: CGO mattn is faster and battle-tested but needs a cross-compile toolchain per target; an embedded server DB (Postgres-in-a-box, rqlite) is overkill for a local sidecar; bbolt/embedded-KV is simpler but loses SQL and migrations. Trade-off: right call for a distributable desktop daemon — but validate modernc's WAL + synchronous semantics match the C library's (the durability story in H2 hinges on it behaving exactly like upstream SQLite).

D3 · Single writer (SetMaxOpenConns(1)) + optimistic revision CAS

The bet: serialize all DB access through one connection and use the revision column as the concurrency arbiter — simple, and it sidesteps WAL's single-writer rule entirely.

Alternatives: a small writer pool with retry-on-SQLITE_BUSY (more read concurrency, but reintroduces the CAS race L2 unless the write is made self-arbitrating); pessimistic BEGIN IMMEDIATE locks (clearer intent, more contention). Trade-off: the single-conn choice makes correctness easy to reason about now, but bottlenecks all reads behind the writer and hides the CAS race behind one config line. If read load ever grows, prefer a read-only second pool over relaxing the writer.

D4 · Polling consumer (100 ms) rather than fs-notify

The bet: dependency-free, predictable, sub-human-perceptible latency.

Alternatives: fsnotify (lower latency, but adds a CGO-ish dep and cross-platform edge cases) or an in-process channel push from publisher to consumer (no file round-trip at all — but then the JSONL log is pure durability, not the transport). Trade-off: 100 ms polling is fine for a supervisory UI; the in-process-channel option is worth considering since publisher and consumer already share a process — it would also sidestep the torn-read (M1) and rescan (L3) concerns.

D5 · Metadata off the canonical/CDC path

The bet: high-churn, low-stakes fields (branch, paths) shouldn't bump revision or generate events.

Alternatives: fold metadata into the canonical record (uniform stream, but noisier CDC and revision churn) or emit a separate lightweight metadata_changed event (FE sees changes without polluting the canonical seq). Trade-off: the current split keeps the canonical stream clean but creates a two-channel read model (M2) the FE must understand. Acceptable if documented; a foot-gun if not.

D6 · In-process synchronous Broadcaster fan-out

The bet: a minimal seam now; let each transport add its own buffering inside its callback.

Alternatives: per-subscriber buffered channels with a drop/disconnect policy (bounded backpressure, slow-subscriber isolation) or a durable per-subscriber queue (survives transport reconnect). Trade-off: the current design is the smallest thing that compiles, but it pushes all backpressure and fault-isolation onto subscribers that don't exist yet — and as written, one panicking subscriber halts delivery (H1). Add recover() + a bounded buffer before the first real transport lands.

09 Recommended improvements

PR-level (before this lands, or as immediate follow-ups)

  • P0 Wrap each broadcaster callback in recover() and give subscribers a bounded buffer with an explicit drop/disconnect policy (H1, D6). This is the one defect that can silently stop the whole pipeline.
  • P0 Add a janitor test suite — watermark math, "never delete an unacked row," and the empty/late-consumer case (H3). The PR claims janitor coverage that the diff does not contain.
  • P1 Resolve the cross-store durability question (H2): either synchronous(FULL) on the canonical path, or a written guarantee that all canonical state is re-derivable and the JSONL log is advisory. Add a crash-injection integration test.
  • P1 Seed each consumer's offset row at startup (at its snapshot seq) so it counts toward the janitor watermark before it can lose events (H3).
  • P1 On a corrupt/unparseable line, trigger a snapshot resync instead of skipping forever (M1); add a malformed-line test.
  • P2 Make the revision write self-arbitrating (rely on row-count from WHERE revision = ?, not the prior read) so L2 doesn't depend on SetMaxOpenConns(1); add a comment binding the two.
  • P2 Document the two-channel read model (canonical CDC vs. pull-only metadata) in the ports doc (M2). Persist or comment the in-memory byte cursor (L3).

Rewrite-level (program-wide)

  • P0 Define and implement a change_log/outbox retention policy before the table grows unbounded; encode it in FK constraints (M3) so the cleanup job can't orphan rows.
  • P1 Stand up the first real CDC consumer (a WS/SSE transport) — it is the only thing that will exercise backpressure, multi-consumer, and the broadcaster contract under real conditions. Much of this review is theoretical until a transport subscribes.
  • P1 Build the SCM-facts producer (poller) — the LCM is a reducer waiting for facts that nothing currently emits; without it the durable lifecycle never transitions in production.
  • P2 Decide whether the runtime-loadable plugin ecosystem (AO's biggest capability lead) is a goal; if so, the compiled-in adapter model needs a registry/discovery story.
  • P2 Port AO's Windows process-teardown / pty lessons before the process runtime lands — it is the area AO paid the most to get right.
  • P2 Add a metrics/health surface; the change_log already gives you a free audit trail to build dashboards on.

10 Open questions for the rewrite author

Is synchronous=NORMAL a deliberate choice with the understanding that the fsync'd JSONL log can outrun the recovered DB on power loss (H2)? What is the intended durability contract — re-derivable-everything, or no-lost-commits?

Is multi-consumer a real near-term goal, or is consumer_offsets keyed defensively? The janitor watermark (H3) is only safe for the single-consumer case as written — which is it designed for?

What is the retention plan for change_log and outbox? Both grow unbounded today; the FK shape (M3) needs to know whether sessions/log rows are ever hard-deleted.

Will the first transport buffer inside its broadcaster callback, or should the broadcaster own per-subscriber queues? The "subscribers must not block" contract (D6) is unenforceable for a network transport.

How does the frontend learn about metadata changes (branch, workspacePath) given they never hit the CDC stream (M2)? Is there a planned read API, or should metadata move onto the stream?

The PR's test plan lists "janitor vacuum" as covered, but I find no janitor test in the diff. Was it dropped, or is it expected in a follow-up?

Is the compiled-in adapter model the intended end state, or a stepping stone toward something like AO's runtime plugin registry? It shapes how SCM/tracker/notifier slots get wired.
