# Apple Watch POC — design

**Status:** proposed
**Date:** 2026-07-21
**Branch:** `worktree-watch-app-poc`

## Goal

A proof-of-concept watchOS app that lets the user *use AO from their wrist*: view
live sessions and send input (typed or dictated) to a running agent, and spawn a
new task by dictating a prompt. It is an interactive POC, not a polished product.

## Non-goals (YAGNI)

- No live terminal / streaming output on the watch (the phone app owns that).
- No PR actions, Kanban board, orchestrator screen, or previews.
- No push notifications or complications (possible future work, out of scope now).
- No QR pairing (the watch has no camera).
- No WatchConnectivity / phone relay (see Architecture).

## Architecture

**A self-contained native SwiftUI watchOS app that talks directly to the AO daemon
over the LAN, using the same HTTP API the phone app uses.** No dependency on the
React Native phone app and no native bridge — the watch app is independent Swift.

```
Apple Watch (SwiftUI)  ──HTTP + Bearer──▶  AO daemon LAN listener
   URLSession                                0.0.0.0:3011 (opt-in, bearer password)
```

Rationale: the phone-relay alternative (watch ⇄ phone via WatchConnectivity, phone
forwards to daemon) would require adding a native WatchConnectivity module to the
Expo/RN app — significant work — and only functions when the phone is nearby.
Direct-to-daemon is the smallest thing that fully delivers the POC goal.

### API contract (already exists in the daemon; mirrors `packages/mobile/lib/api.ts`)

- **List sessions:** `GET /api/v1/sessions` → `{ sessions: WireSession[] }`
  (fields used: `id`, `displayName`, `status`, `activity`, `isTerminated`).
- **Send input to a session:** `POST /api/v1/sessions/{id}/send` body `{ "message": string }`.
- **Spawn a task:** `POST /api/v1/sessions` body
  `{ projectId, prompt, harness?, kind: "worker" }`.
- **List projects (for spawn target):** `GET /api/v1/projects` → `{ projects: [{id,name}] }`.
- **Auth:** header `Authorization: Bearer <password>` on every request.
- **Base URL:** `http://<host>:<port>` — LAN listener default port **3011**
  (configurable). Plaintext HTTP by design (trusted-home-network tool).

The daemon returns an error envelope `{ error, code, message, requestId }` on
non-2xx; the watch surfaces `message` when present.

## Components / screens

Three screens, each with one clear job:

1. **Connect (settings).** Text fields for host, port (default `3011`), and
   password, persisted in `UserDefaults` (POC-grade; not the Keychain — see Risks).
   watchOS text fields provide dictation/Scribble for free. A "Test" button calls
   `GET /sessions` and reports reachable/unreachable. For convenience the fields can
   be pre-seeded with the user's current values so first run needs no typing.
2. **Sessions.** A `List` of live sessions from `GET /sessions`, showing display
   name + status/activity, with pull-to-refresh. Tap a row → Session screen. A
   toolbar "＋" opens the Spawn flow.
3. **Session.** Shows the session identity, a text field ("Speak or type…") that
   dictates/types a message, and a **Send** button → `POST /sessions/{id}/send`
   with haptic confirmation. The **Spawn** flow is the same input bound to
   `POST /sessions` with a chosen project.

### Networking module

A single `AOClient` struct/actor: holds config, builds requests, adds the bearer
header, applies a **short request timeout** (~10s) so an unreachable Mac never
hangs the watch UI, decodes JSON with `Codable`, and maps the error envelope to a
thrown `AOError`. All screens call through it. This is the one unit that touches
the network and is independently testable with a stubbed `URLProtocol`.

### App Transport Security

The daemon is plaintext HTTP, so the watch app's `Info.plist` needs an ATS
exception (`NSAppTransportSecurity` → `NSAllowsArbitraryLoads = true` for the POC,
or a scoped exception for the LAN host). Without it, `URLSession` blocks the
cleartext request. This is injected via the config plugin (below).

## Durability — surviving Expo prebuild

`packages/mobile/ios/` is gitignored and regenerated by `expo prebuild` /
`expo run:ios`, which would erase any hand-added Xcode target. To make the watch
target durable and committed:

- Add the **`@bacons/apple-targets`** Expo config plugin.
- Store the watch app's Swift source + `Info.plist` + assets in a **tracked**
  folder: `packages/mobile/targets/watch/`.
- The plugin regenerates the watchOS target into `ios/` on every prebuild, so the
  dev loop (`expo run:ios`) no longer wipes it.
- Register the plugin in `packages/mobile/app.json`.

**Fallback if the plugin is incompatible with Expo SDK 54 / RN 0.81:** keep the
Swift source in the same tracked `targets/watch/` folder and add a small
documented script + README steps to re-add the Xcode target after a prebuild. The
watch source stays committed either way; only the re-attachment mechanism differs.
This fallback is only taken if the plugin approach fails verification.

## What I can do vs. what the user must do

**I (assistant) will:** create the tracked watch source, the `AOClient`, the three
screens, the ATS/Info.plist, wire the config plugin into `app.json`, run
`expo prebuild` to verify the target regenerates, and build the watch scheme with
`xcodebuild` to confirm it compiles.

**The user must (device/GUI only, cannot be automated here):**
- Set the signing **Team** on the watch target in Xcode (needs their Apple ID).
- Enable **Developer Mode** on the watch and approve device-trust prompts.
- Select the watch scheme + Apple Watch destination and **Build & Run** to install.
- Re-sign weekly if on a free Apple ID (7-day provisioning profile).

## Testing / verification

- **Unit:** `AOClient` request building + error mapping via a stubbed `URLProtocol`
  (no live daemon needed).
- **Build:** `expo prebuild` regenerates the watch target; `xcodebuild` compiles the
  watch scheme for the watchOS simulator.
- **Manual (user):** with Connect Mobile enabled on the desktop, the watch lists
  sessions, sends a dictated message that appears in the desktop session, and
  spawns a task.

## Risks / open caveats

- **Reachability:** the watch reaches a `192.168.x` / Tailscale host only when near
  the phone (proxying through it) or directly on the same Wi-Fi. Off on cellular →
  no reach. Inherent to a LAN tool; documented, not fixed.
- **Password in `UserDefaults`:** POC-grade. The phone app deliberately keeps the
  bearer password in the Keychain. For a POC on a personal device this is an
  accepted shortcut; hardening to Keychain is noted as follow-up.
- **Plugin compatibility:** `@bacons/apple-targets` on SDK 54 is the main unknown;
  the fallback above de-risks it.
